## 一、文档概述

本 PRD 聚焦 CardScope 的**核心功能链路**：用户拍摄体育球星卡照片 → OCR/AI 识别卡片 → 返回价格数据 → AI 品相评级。这是产品最核心的价值闭环，也是用户留存和付费转化的关键路径。

### 文档范围

| 包含 | 不包含 |
|------|--------|
| 拍照/选图入口 | 用户账号体系 |
| OCR 文字提取 | 社区交易市场 |
| AI 卡片匹配识别 | 推送通知系统 |
| 价格数据获取与展示 | 收藏管理详细逻辑 |
| AI 品相评级分析 | 订阅付费详细逻辑 |
| 扫描结果展示 | |

## 二、目标与成功指标

### 2.1 产品目标

1. **核心体验**：用户从拍照到看到卡片估值 < 3 秒
2. **识别准确率**：在光线良好条件下，卡片识别命中率 ≥ 85%
3. **评级可信度**：AI 评级结果与专业评级（PSA/BGS）偏差 ≤ 1.0 分
4. **付费转化**：通过扫描体验驱动 Pro 订阅转化率 ≥ 8%

### 2.2 关键指标（KPI）

| 指标 | 定义 | 目标值 |
|------|------|--------|
| 扫描成功率 | 成功识别的扫描 / 总扫描次数 | ≥ 85% |
| 平均识别耗时 | 从拍照到展示结果的平均时间 | < 3s |
| 扫描→详情转化 | 查看完整卡片详情的用户比例 | ≥ 70% |
| 扫描→收藏转化 | 扫描后添加收藏的用户比例 | ≥ 30% |
| 评级使用率 | 查看详情后点击评级的用户比例 | ≥ 40% |
| 日均扫描次数 | 活跃用户日均扫描量 | ≥ 2.5 |

### 4.2 OCR 文字提取详细设计

#### 技术方案：Apple Vision Framework

| 配置项 | 值 | 说明 |
|--------|---|------|
| API | `VNRecognizeTextRequest` | iOS 13+ 系统级 OCR |
| 识别级别 | `.accurate` | 准确模式（非快速模式），提升识别精度 |
| 语言校正 | `usesLanguageCorrection = true` | 启用上下文语言校正 |
| 最小文字高度 | `0.02` | 相对于图片高度的最小文字比例 |
| 候选数 | `topCandidates(1)` | 每个观察取置信度最高的1个候选 |

#### OCR 输出处理

```
原始 OCR 输出（多行拼接）
    │
    ▼
"PANINI PRIZM 2023-24  LeBron James  #1  Los Angeles Lakers"
    │
    ▼ normalizedText()
"panini prizm 2023 24 lebron james 1 los angeles lakers"
    │
    ▼ normalizedTokens()
["panini", "prizm", "2023", "24", "lebron", "james", "los", "angeles", "lakers"]
```

#### OCR 性能要求

| 指标 | 要求 |
|------|------|
| 文字提取延迟 | < 800ms（iPhone 12 及以上） |
| 支持语言 | English（球星卡主要为英文） |
| 识别准确率 | 标准印刷体 ≥ 95%，特殊字体 ≥ 80% |

#### 当前局限与改进方向

| 局限 | 影响 | V2 改进方案 |
|------|------|------------|
| 仅文字识别，无图像识别 | 相同球员不同卡面难以区分 | 引入 Vision 图像分类/CoreML 模型 |
| 无卡片区域检测 | 背景文字可能干扰 | 增加 VNDetectRectangles 预处理，先定位卡片再 OCR |
| 依赖文字清晰度 | 老旧/磨损卡片识别率低 | 图像增强预处理 + 多帧融合 |
| 无法识别卡背信息 | 部分卡片正面信息不足 | 支持正反面双拍模式 |

### 4.3 卡片匹配算法详细设计

#### 多维度评分机制

```
总分 = 球员名匹配(0/4) + 品牌匹配(0/2) + 系列匹配(0/2) + 卡号匹配(0/2)
     + 年份匹配(0/1) + 平行版匹配(0/1) + Token重叠加分(0~2)
```

| 维度 | 匹配方式 | 分值 | 权重理由 |
|------|---------|------|----------|
| 球员名 (player_name) | 全名子串包含 | +4 | 最具区分度的字段 |
| 品牌 (brand) | 子串包含 | +2 | 高区分度，如 Panini vs Topps |
| 系列 (set_name) | 子串包含 | +2 | 同品牌下区分系列 |
| 卡号 (card_number) | 子串包含 | +2 | 精确匹配辅助 |
| 年份 (year) | 子串包含 | +1 | 区分不同年份版本 |
| 平行版 (parallel) | 子串包含 | +1 | 区分基础版和特殊版 |
| Token 重叠 | 球员名分词交集 | +0~2 | 容错：OCR 部分缺失时的补偿 |

**最高可能分数**：14 分
**命中阈值**：≥ 2 分（当前设定，平衡召回率与精确率）

#### 匹配策略

```
1. 获取全量卡片库（本地 + 远程）
2. 对每张卡片逐一评分
3. 按分数降序排列
4. 取最高分候选
5. 若最高分 ≥ 2 → 命中，返回该卡片
6. 若最高分 < 2 → 未命中，返回空结果
```

#### 评分阈值策略

| 分数区间 | 判定 | 置信度 | 用户体验 |
|---------|------|--------|----------|
| ≥ 8 | 强匹配 | 极高 | 直接展示结果 |
| 4 ~ 7 | 中等匹配 | 中高 | 展示结果 + "可能是这张卡" |
| 2 ~ 3 | 弱匹配 | 较低 | 展示结果 + "请确认是否正确" |
| < 2 | 未匹配 | — | 展示未识别页面 |

> **V2 改进**：分级置信度展示，让用户可以从多个候选中选择正确卡片。

#### 匹配数据源优先级

```
1️⃣ Supabase 远程查询（1,792+ 张卡片）
    ↓ 失败/超时
2️⃣ SwiftData 本地缓存
    ↓ 未命中
3️⃣ Mock 数据兜底
```

### 4.5 AI 品相评级详细设计

#### 评级概述

AI 评级功能通过计算机视觉技术，对卡片照片进行四个维度的品相分析，输出类 PSA 评级分数（6.0~10.0），帮助用户在送评前预判卡片品相。

> **权限控制**：AI 评级为 **Pro 专属功能**，免费用户点击评级按钮将触发 Paywall。

#### 四维评分体系

```
┌────────────────────────────────────────────────────┐
│                    AI Grade 评级总分                  │
│                                                     │
│    Overall = Centering × 0.30                       │
│            + Corners   × 0.25                       │
│            + Edges     × 0.20                       │
│            + Surface   × 0.25                       │
│                                                     │
│    分值范围: 6.0 ~ 10.0                              │
└────────────────────────────────────────────────────┘
```

##### 维度 1: 居中度 (Centering) — 权重 30%

| 项目 | 说明 |
|------|------|
| 技术 | `VNDetectRectanglesRequest` 检测卡片矩形边界 |
| 算法 | 计算检测到的矩形中心点与图像中心点 (0.5, 0.5) 的偏移量 |
| 公式 | `drift = min(1.0, (abs(cx-0.5) + abs(cy-0.5)) × 2.5)` |
| 评分 | `score = max(6.5, 10.0 - drift × 3.0)` |
| 最低置信度 | 0.55（低于此值使用默认分 8.8） |

**居中度等级参考**：

| 偏移量 | 评分范围 | 对应 PSA 等级 |
|--------|---------|--------------|
| < 0.05 | 9.5-10.0 | Gem Mint |
| 0.05-0.15 | 8.5-9.5 | Mint / Near Mint-Mint |
| 0.15-0.30 | 7.0-8.5 | Near Mint / Excellent-Mint |
| > 0.30 | 6.5-7.0 | Excellent 及以下 |

##### 维度 2: 边角 (Corners) — 权重 25%

| 项目 | 说明 |
|------|------|
| 技术 | `CIAreaAverage` 滤镜计算全图平均亮度 |
| 算法 | 亮度偏差分析，偏离标准值越大品相越差 |
| 公式 | `score = max(6.8, min(10.0, 9.7 - (brightness - 0.55) × 1.4))` |
| 原理 | 边角磨损会改变光散射特性，影响整体亮度分布 |

> **V2 改进**：裁剪卡片四角区域单独分析，使用 edge detection 检测磨损痕迹。

##### 维度 3: 边缘 (Edges) — 权重 20%

| 项目 | 说明 |
|------|------|
| 技术 | `CIFilter.edges()` 边缘检测滤镜 |
| 算法 | 计算边缘检测结果的平均亮度（对比度） |
| 公式 | `score = max(6.5, min(10.0, 7.0 + edgeContrast × 3.2))` |
| 原理 | 清晰锐利的边缘 = 高对比度 = 高分 |

##### 维度 4: 表面 (Surface) — 权重 25%

| 项目 | 说明 |
|------|------|
| 技术 | `CIAreaAverage` + `CIAreaMinimum` + `CIAreaMaximum` |
| 算法 | 计算亮度方差 = (max - min) × (1 - |avg - 0.5|) |
| 公式 | `score = max(6.2, min(10.0, 9.8 - variance × 2.8))` |
| 原理 | 表面均匀 = 低方差 = 高分；划痕/污渍 = 高方差 = 低分 |

#### 评级结果输出

```swift
GradeBreakdown {
    centering: Double  // 6.0 ~ 10.0
    corners: Double    // 6.0 ~ 10.0
    edges: Double      // 6.0 ~ 10.0
    surface: Double    // 6.0 ~ 10.0

    // 计算属性
    overall: Double    // 加权平均
    letterGrade: String // "Gem Mint" / "Mint" / "NM-MT" / ...
}
```

| 综合得分 | 等级名称 | 颜色标识 | 对应 PSA |
|---------|---------|---------|---------|

| 9.0-9.4 | Mint | 🟢 绿色 | PSA 9 |

struct GradeBreakdown {
    let centering: Double  // 30% 权重

## 七、状态管理与错误处理

### 7.1 扫描状态机

```
           ┌──────────┐
           │  idle     │ ← 初始状态 / 完成后重置
           └────┬─────┘
                │ 用户触发扫描
                ▼
           ┌──────────┐
           │ capturing │ ← 相机拍照中 / 选择图片中
           └────┬─────┘
                │ 获得图片
                ▼
           ┌──────────┐
           │ analyzing │ ← OCR + 匹配处理中（显示 loading）
           └────┬─────┘
                │
         ┌──────┴──────┐
         ▼             ▼
    ┌─────────┐   ┌──────────┐
    │ found   │   │ notFound │
    │ 展示结果  │   │ 展示引导  │
    └─────────┘   └──────────┘
```

### 7.2 错误处理策略

| 错误场景 | 处理方式 | 用户提示 |
|---------|---------|---------|
| 相机权限被拒 | 展示静态背景 + 引导前往设置 | "请在设置中允许相机权限" |
| OCR 提取为空 | 返回 notFound 状态 | "未能识别卡片文字，请确保光线充足" |
| 网络请求超时 | 降级至本地数据 | 静默降级，不打断用户流程 |
| Supabase 不可达 | 使用 SwiftData 缓存 | 详情页标注"离线数据" |
| 价格获取全部失败 | 使用卡片库内置价格 | 标注"参考价格" |
| 评级图片无法分析 | 返回默认评分 (8.8/8.6/8.5/8.4) | 正常展示，不暴露错误 |

### 7.3 离线模式

```
网络不可用时的降级策略：

扫描识别：正常工作（OCR 在设备端执行）
卡片匹配：使用 SwiftData 本地缓存库 → Mock 数据
价格数据：使用卡片内置价格字段 + Mock 历史走势
AI 评级：正常工作（所有计算在设备端执行）

标注："数据更新于 {lastSyncDate}"
```

## 十三、风险与缓解

| 风险 | 影响 | 概率 | 缓解措施 |
|------|------|------|---------|
| OCR 准确率不足 | 识别成功率低，用户流失 | 中 | V2 引入多模态 AI 视觉识别 |
| 价格数据不准 | 用户信任度下降 | 中 | 多数据源交叉验证 + 标注数据置信度 |
| AI 评级与真实评级差异大 | 用户质疑功能价值 | 高 | 明确标注"AI 参考评级"，V2 训练专用模型 |
| 卡片库覆盖不全 | 新卡/小众卡无法识别 | 中 | 持续扩充数据库 + 用户反馈补录机制 |
| Apple Vision API 变更 | 需要适配新版本 | 低 | 抽象 OCR 接口层，便于切换实现 |
